What is Infrastructure as Code?

Infrastructure as Code means describing your servers, networks, and databases in text files — just like application code. Instead of clicking around the AWS console, you write a `.tf` file and Terraform creates the infrastructure for you.

Manual vs IaC — the fundamental difference
The 6 Core Building Blocks
Terraform reads your config + state → calls provider APIs → creates/modifies real infrastructure
📋

Provider

A plugin that translates Terraform resources into real API calls. The AWS provider turns aws_instance into EC2 API calls. Over 3,000 providers exist on the registry.

🏗️

Resource

The fundamental building block. A resource block declares one infrastructure object — EC2, S3 bucket, VPC. Terraform creates, updates, or destroys it to match your config.

📦

State File

A JSON file tracking every resource Terraform manages. It maps your config to real-world IDs. Without state, Terraform can't know what already exists or what needs changing.

🔍

Data Source

Read-only queries to fetch existing infrastructure. Use data blocks to look up an AMI ID, an existing VPC, or a secret — any external value your resources need.

🧩

Module

A directory of .tf files with inputs and outputs — the reusable component pattern of Terraform. Write once, call many times with different parameters.

🌍

Workspace

Isolated state environments within the same config. Use workspaces to maintain separate state for dev/staging/prod without duplicating your code.

The Core Terraform Workflow

Every Terraform operation follows the same 4-command cycle. Mastering when and why to use each command is the foundation of everything else.

The core Terraform loop — always preview with plan before you apply
```
Each Command Explained
1

terraform init

Downloads the provider plugins declared in your config into a local .terraform/ directory. Initialises the backend. Must be run first, and again after any provider version change. Safe to run multiple times — it's idempotent.

2

terraform validate

Checks your HCL syntax and internal consistency — catches typos, missing required arguments, and invalid references. Makes no API calls. Free to run as many times as you like. Use as a pre-commit hook.

3

terraform plan

The most important command. Compares your config against state, calls provider APIs to check real-world status, and produces a diff: + to add, ~ to change, - to destroy. Use -out=tfplan to save it for apply.

4

terraform apply

Executes the plan. Without -auto-approve it shows the plan again and asks for confirmation. In CI/CD, always use a saved plan file: terraform apply tfplan — this guarantees you apply exactly what you reviewed.

5

terraform destroy

Destroys all resources managed by this config. Use -target to destroy specific resources. Protected by lifecycle { prevent_destroy = true } on critical resources.

Pro tip: Always use terraform plan -out=tfplan then terraform apply tfplan. This ensures the apply executes exactly the plan you reviewed — no surprise changes if the cloud state changed between plan and apply.
HCL — HashiCorp Configuration Language

HCL is Terraform's native language. It's designed to be human-readable and machine-parseable. Everything you write in Terraform is HCL. Learn these patterns and you can write any Terraform config.

Anatomy of a resource block — block type, resource type, local name, arguments
Provider Configuration
provider.tfHCL
terraform { required_version = ">= 1.6.0" # pin Terraform version required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" # ~> means >= 5.0, < 6.0 } } } provider “aws” { region = var.aws_region profile = “my-aws-profile” # or use env vars / OIDC default_tags { # applied to ALL resources tags = { ManagedBy = “Terraform” Environment = var.environment Team = “platform-sre” } } }
Variables — Inputs to your config
variables.tfHCL
# Simple string variable variable "environment" { description = "Deployment environment (dev/staging/prod)" type = string default = "dev" validation { condition = contains([“dev”, “staging”, “prod”], var.environment) error_message = “Must be dev, staging, or prod.” } } # Object variable — grouped config variable “instance_config” { type = object({ type = string count = number spot = bool }) default = { type = “t3.micro” count = 2 spot = false } } # Sensitive variable — won’t print in logs variable “db_password” { type = string sensitive = true }
Variable precedence (highest wins): -var flag → TF_VAR_ env var → terraform.tfvars → *.auto.tfvars → default value in variable block
for_each — Create multiple resources from a list/map
for_each patternsHCL
# Create 3 S3 buckets from a set resource "aws_s3_bucket" "envs" { for_each = toset(["dev", "staging", "prod"]) bucket = "myapp-${each.key}-data" } # Access: aws_s3_bucket.envs["dev"].arn # Create resources from a map (more powerful) variable “ec2_instances” { default = { web = { type = “t3.small”, count = 2 } api = { type = “t3.medium”, count = 3 } worker = { type = “t3.large”, count = 1 } } } resource “aws_instance” “fleet” { for_each = var.ec2_instances ami = data.aws_ami.ubuntu.id instance_type = each.value.type # each.key = “web”, “api”, “worker” }
Lifecycle — Control resource behaviour
lifecycle meta-argumentHCL
resource "aws_db_instance" "prod_db" { # ... other config ... lifecycle { # Prevent accidental deletion of the production DB prevent_destroy = true ``` # Don't let Terraform change the password (managed by Secrets Manager) ignore_changes = [password, snapshot_identifier] # Create new resource BEFORE destroying old one (zero-downtime) create_before_destroy = true ``` } }
Understanding Terraform State

State is the most misunderstood part of Terraform — and the most important. The state file is a JSON document that records every resource Terraform manages, mapping your config to real cloud resource IDs.

Terraform compares desired config against state + real cloud to calculate what needs to change
```
Remote Backend — S3 + DynamoDB

Never store state locally on your laptop for team work. Use a remote backend with locking to prevent two people running apply simultaneously and corrupting state.

Both engineers share state via S3, DynamoDB prevents concurrent apply runs
```
backend.tfHCL
terraform { backend "s3" { bucket = "mycompany-terraform-state" key = "prod/networking/terraform.tfstate" region = "ap-south-1" encrypt = true # AES-256 at rest dynamodb_table = "terraform-state-locks" # for locking kms_key_id = "arn:aws:kms:..." # optional: CMK } } # After adding this, run: terraform init -reconfigure
State Commands You Must Know
CommandPurposeWhen to use
terraform state listList all resources in stateAudit what Terraform manages
terraform state show aws_instance.webShow all attributes of one resourceDebug, find IDs, understand current state
terraform import aws_instance.web i-0abc123Import existing cloud resource into stateBring manually-created infra under TF management
terraform state mv old.resource new.resourceRename/move resource in stateRefactoring — avoid destroy+recreate
terraform state rm aws_instance.webRemove from state without destroyingStop managing a resource (keep it alive)
terraform apply -replace=aws_instance.webForce recreation of specific resourceReplace a corrupted/broken instance
⚠ Warning: Never manually edit the tfstate file. If state gets corrupted, use terraform state commands or restore from your S3 bucket's versioned backup. Always enable S3 versioning on your state bucket.
Modules — The Reuse Pattern of Terraform

A module is just a directory containing .tf files. Every Terraform config is already a module — the "root module". Child modules let you package infrastructure as reusable components with defined inputs and outputs.

Root module orchestrates child modules — each module has defined inputs (variables) and outputs
```
Writing Your First Module
modules/vpc/main.tfHCL
# modules/vpc/main.tf — the module definition resource “aws_vpc” “this” { cidr_block = var.cidr enable_dns_hostnames = true tags = { Name = ”${var.name}-vpc” } } resource “aws_subnet” “public” { for_each = toset(var.public_subnets) vpc_id = aws_vpc.this.id cidr_block = each.value availability_zone = element(var.azs, index(var.public_subnets, each.value)) }
modules/vpc/variables.tf + outputs.tfHCL
# variables.tf — module inputs variable "name" { type = string } variable "cidr" { type = string } variable "azs" { type = list(string) } variable "public_subnets" { type = list(string) } # outputs.tf — what callers can use output “vpc_id” { value = aws_vpc.this.id } output “subnet_ids” { value = [for s in aws_subnet.public : s.id] }
main.tf — calling the moduleHCL
# Call the same module 3 times with different inputs module "vpc_dev" { source = "./modules/vpc" name = "dev" cidr = "10.0.0.0/16" azs = ["ap-south-1a", "ap-south-1b"] public_subnets = ["10.0.1.0/24", "10.0.2.0/24"] } module “vpc_prod” { source = ”./modules/vpc” name = “prod” cidr = “10.1.0.0/16” azs = [“ap-south-1a”, “ap-south-1b”] public_subnets = [“10.1.1.0/24”, “10.1.2.0/24”] } # Use outputs from the module output “prod_vpc_id” { value = module.vpc_prod.vpc_id }
Top tip: Before writing your own module, check registry.terraform.io. The terraform-aws-modules/vpc/aws module is battle-tested by thousands of companies and handles edge cases you'd miss. Use community modules for standard infra, write custom ones for your organisation-specific patterns.
Terraform in CI/CD Pipelines

Running Terraform manually from laptops doesn't scale. In a team, every infrastructure change should go through a pull request, be reviewed, and be applied automatically by a CI/CD system — with a human approval gate for production.

Full CI/CD pipeline — plan on PR, apply on merge, OIDC auth means zero stored AWS credentials
```
GitHub Actions Workflow
.github/workflows/terraform.ymlYAML
name: Terraform CI/CD on: pull_request: paths: [‘infra/**’] # only run when .tf files change push: branches: [main] permissions: id-token: write # required for OIDC contents: read pull-requests: write # to post plan comment on PR jobs: terraform: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 ``` - name: Configure AWS via OIDC # no stored keys! uses: aws-actions/configure-aws-credentials@v4 with: role-to-assume: arn:aws:iam::123456:role/GitHubActionsRole aws-region: ap-south-1 - uses: hashicorp/setup-terraform@v3 with: { terraform_version: '1.7.0' } - run: terraform init - name: Terraform Plan id: plan run: terraform plan -out=tfplan -no-color 2>&1 | tee plan.txt - name: Post Plan to PR # only on PRs if: github.event_name == 'pull_request' uses: actions/github-script@v7 with: script: | github.rest.issues.createComment({ ...context.repo, issue_number: context.issue.number, body: '```\n' + require('fs').readFileSync('plan.txt', 'utf8') + '```' }) - name: Terraform Apply # only on merge to main if: github.ref == 'refs/heads/main' run: terraform apply -auto-approve tfplan
```
BFSI context: In regulated environments (DORA, SOX), you need an audit trail of every infrastructure change. This pipeline provides it automatically — every plan and apply is logged in GitHub Actions, linked to a PR, and associated with a named reviewer. That's your change evidence for auditors.

Best Practices Quick Reference
✓ Do

Always use remote backend with state locking

Never store state locally in a team environment. S3 + DynamoDB is the standard for AWS. Enable versioning on the S3 bucket so you can restore state if it gets corrupted.

✗ Don't

Never hardcode credentials in .tf files

Hardcoding AWS keys in .tf files is a critical security risk that will get caught in audits. Use OIDC for CI/CD, IAM roles for EC2/Lambda, and AWS profiles for local development.

✓ Do

Pin provider and module versions

Always specify version constraints (~> 5.0 for providers). Unpinned versions auto-upgrade on terraform init -upgrade and can silently break your infrastructure.

💡 Tip

Standard file structure: one file per concern

main.tf resources · variables.tf inputs · outputs.tf exports · provider.tf provider · backend.tf state. Every Terraform engineer knows this layout.

✓ Do

Use lifecycle blocks to protect critical resources

Add prevent_destroy = true to databases, S3 buckets with data, and KMS keys. Add ignore_changes for attributes managed outside Terraform.

💡 Tip

Run fmt + validate as pre-commit hooks

Install the pre-commit framework with terraform hooks. Catches formatting, syntax, and common mistakes before they hit your repo — saving CI minutes and code review time.
