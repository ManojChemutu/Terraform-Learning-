Core concepts you must master
📋

Providers

Plugins that allow Terraform to interact with APIs. AWS, Azure, GCP, Kubernetes — each provider must be declared and initialised with `terraform init`.

🏗️

Resources

The fundamental unit. Each `resource` block describes one infrastructure object — an EC2 instance, S3 bucket, VPC. Resources have type, name, and arguments.

📦

State File

Terraform's map of reality. The `terraform.tfstate` JSON file tracks what Terraform has created. It is the source of truth for drift detection and plan generation.

🔁

The Core Workflow

`init` → `plan` → `apply` → `destroy`. Init downloads providers. Plan previews changes. Apply creates/modifies infra. Destroy tears it down.

🧩

Modules

Reusable, composable infrastructure components. A module is just a directory of `.tf` files with inputs and outputs — the building block of scalable Terraform codebases.

📊

Dependency Graph

Terraform builds a directed acyclic graph of all resources. It resolves implicit dependencies (via references) and explicit ones (`depends_on`) to determine creation order.

🔍

Data Sources

Read existing infrastructure not managed by this config. Use `data` blocks to fetch AMI IDs, VPC IDs, secrets — any external data your resources need.

🔒

Remote Backend

Store state in S3 (or Terraform Cloud) instead of locally. Combined with DynamoDB for locking — prevents two engineers from running `apply` simultaneously.

🌍

Workspaces

Isolate state within the same config. Use workspaces to maintain separate state for dev/staging/prod. Accessed via `terraform workspace new` / `select`.

Provider Configuration
provider.tf HCL
terraform { ``` required_providers { aws = { source = “hashicorp/aws” version = ”~> 5.0” } } required_version = ”>= 1.6.0” } provider “aws” { region = var.aws_region profile = “default” }
```
Resource Block
main.tf — EC2 Instance HCL
resource "aws_instance" "web" { ``` ami = data.aws_ami.ubuntu.id instance_type = var.instance_type tags = { Name = ”${var.env}-web-server” Environment = var.env ManagedBy = “Terraform” } lifecycle { create_before_destroy = true ignore_changes = [ami] } }
```
Variables & Outputs
variables.tf + outputs.tf HCL
# variables.tf ``` variable “instance_type” { description = “EC2 instance type” type = string default = “t3.micro” validation { condition = contains([“t3.micro”,“t3.small”,“t3.medium”], var.instance_type) error_message = “Must be t3.micro, t3.small, or t3.medium.” } } # outputs.tf output “instance_public_ip” { description = “Public IP of the web instance” value = aws_instance.web.public_ip sensitive = false }
```
Remote Backend (S3 + DynamoDB)
backend.tf HCL
terraform { ``` backend “s3” { bucket = “my-tf-state-bucket” key = “prod/networking/terraform.tfstate” region = “ap-south-1” encrypt = true dynamodb_table = “terraform-state-lock” } }
```
for_each & Dynamic Blocks
dynamic resources HCL

Production-grade best practices
```
✓ Do

Always use a remote backend with state locking

Never store state locally on your laptop in a team environment. Use S3 + DynamoDB (AWS) or Terraform Cloud. State locking prevents concurrent `apply` runs from corrupting the state file.

✗ Don't

Never hardcode credentials in .tf files

Hardcoding AWS keys, passwords, or secrets directly in `.tf` files is a critical security risk. Use environment variables, AWS profiles, IAM roles (OIDC for CI/CD), or Vault for secret injection.

✓ Do

Pin provider and module versions

Always specify version constraints for providers (`~> 5.0`) and module sources. Unpinned versions will auto-upgrade and silently break your infrastructure with a `terraform init -upgrade`.

💡 Tip

Structure code for readability: one file per concern

Use `main.tf` for resources, `variables.tf` for all inputs, `outputs.tf` for exports, `provider.tf` for provider config, and `backend.tf` for state config. This is the standard layout the whole community follows.

✗ Don't

Don't use `terraform apply` directly in CI/CD without a plan review gate

Always run `terraform plan -out=tfplan` first and fail the pipeline if the plan has destructive changes (`-destroy` count > 0). Require human approval for prod applies using GitHub Environments or Atlantis.

✓ Do

Tag every resource with a standard tag set

Always apply `Environment`, `ManagedBy = "Terraform"`, `Team`, and `CostCenter` tags. Use a `default_tags` block in the AWS provider to apply base tags to all resources without repeating yourself.

💡 Tip

Use `terraform fmt` and `terraform validate` as pre-commit hooks

Install `pre-commit` with hooks for `terraform fmt -check`, `terraform validate`, and `tflint`. This catches syntax errors and drift from code standards before they ever reach your repo.

✗ Don't

Don't put all resources in one giant main.tf

A single 1000-line `main.tf` becomes unmaintainable fast. Split into logical groups — networking, compute, security, IAM — or better yet, separate state per layer to reduce blast radius of a bad `apply`.

✓ Do

Use `lifecycle` blocks to protect critical resources

Add `prevent_destroy = true` to databases, S3 buckets with data, and KMS keys. Add `ignore_changes = [ami]` to EC2 instances patched outside Terraform. Use `create_before_destroy` for zero-downtime replacements.

"Terraform code is infrastructure documentation. Write it as if the next person to read it is a tired on-call engineer at 3am who has never seen your repo — because that person might be you."
